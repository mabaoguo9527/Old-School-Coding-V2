第23章 双亲委派

你不会的事交给上级。上级不会再往上。直到最高层说：这事我管不了。

CHEP后第二十二天。

古法编程班第七天。

第一周的课程进入了最后一天——"综合练习：写一个计算器"。

按照马农的培训计划，今天本该是纯练习。但他改了计划。因为昨天发生了一件事。

昨天是第六天。教异常处理。

林夕在写文件读取练习的时候，代码报了一个错。不是编译错误——而是运行时异常。

``
Exception in thread "main" java.lang.ClassNotFoundException: 
    com.tencent.origin.RuntimeOptimizer
`

ClassNotFoundException。找不到类。

但林夕的代码里根本没有用到任何com.tencent.origin的东西。她只是写了一个简单的文件读取。

刘韧也遇到了同样的问题。然后周明。然后又一个人。

马农看了错误堆栈。

他立刻明白了。

这不是学员的代码问题——是JVM层面的。这些笔记本电脑上的JDK虽然是离线安装的干净版本，但操作系统层面残留着Origin的运行时钩子。Origin在2036年推送了一个系统级补丁，让所有Java进程在启动时自动尝试加载com.tencent.origin.RuntimeOptimizer——一个运行时优化器。

CHEP之前，这个类由Origin的本地代理提供。CHEP之后，代理死了，但钩子还在。

"不用管它。"马农说。"加一个启动参数就行。"

他在白板上写：

`bash
java -Djava.system.class.loader=default -cp . Hello
`

"把Origin的钩子绕过去。"

问题解决了。但赵天问了一个问题：

"马农，JVM加载一个类的时候，到底是怎么找到它的？"

马农看了他一眼。

"明天讲。"

所以今天，马农临时加了一堂课——ClassLoader双亲委派。

这不在第一周的计划里。按照原定安排，ClassLoader应该在第三周"网络编程"之前讲，作为理解JVM类加载机制的基础。但昨天的Origin钩子事件让马农觉得，有些东西不能等。

"不是为了让你们精通ClassLoader。"他站在白板前。"是为了让你们理解——你们写的每一行代码，从.java到运行，中间发生了什么。"

他在白板上画了一条竖线。

"你们写一个Hello.java。"马农开始。

"然后javac Hello.java——编译。生成Hello.class。这是字节码文件。"

"然后java Hello——运行。JVM启动。第一件事：找到Hello这个类，把它的字节码加载到内存里。"

"问题是——JVM怎么找到它的？"

他在竖线旁边画了三个方框，从上到下排列：

`
┌───────────────────────────┐
│   Bootstrap ClassLoader   │  ← 最高级：JDK核心类
│   (启动类加载器)            │     java.lang., java.util.
└───────────┬───────────────┘
            │ 找不到？往下传
┌───────────▼───────────────┐
│   Extension ClassLoader   │  ← 中间级：扩展类
│   (扩展类加载器)            │     javax., jdk.
└───────────┬───────────────┘
            │ 找不到？再往下传
┌───────────▼───────────────┐
│  Application ClassLoader  │  ← 最低级：你写的类
│   (应用类加载器)            │     Hello.class在这里
└───────────────────────────┘
`

"三层。从上到下。"马农说。"当JVM要加载一个类的时候，流程是反过来的——从下往上问。"

他在图旁边画了箭头，方向从下到上：

"Application ClassLoader接到请求：加载Hello。它不会自己先找。它先问上级——Extension ClassLoader，你能加载吗？"

"Extension ClassLoader说：我也不先找。我先问我的上级——Bootstrap ClassLoader，你能加载吗？"

"Bootstrap ClassLoader是最高级。它去自己的管辖范围找——java.lang、java.util这些JDK核心包。Hello不在它的范围里。它说：找不到。"

"于是请求回到Extension ClassLoader。Extension在自己的范围找——扩展包。还是找不到。说：找不到。"

"请求最终回到Application ClassLoader。Application在classpath——你的项目目录——里找。找到了Hello.class。加载。完成。"

他放下笔。

"这就是双亲委派。每一级在自己动手之前，先问上级。"

产品经理举手："为什么要这么麻烦？直接找不就行了？"

马农说了一句后来被学员反复引用的话：

"因为安全。"

他在白板上写了一个例子：

`
假设有人自己写了一个类，也叫 java.lang.String。
如果没有双亲委派——JVM可能加载了假的String。
有了双亲委派——Bootstrap ClassLoader永远先加载JDK自己的String。
你写的假String永远不会被用到。
`

"双亲委派是一种信任链。"马农说。"下级不信任自己，先问上级。最高级最可信。"

他看了一圈学员。

"就像军队。一个士兵接到命令，他不自己决定怎么执行——他先报告班长。班长不确定的报告排长。排长不确定的报告连长。一直往上。直到有人说：这个我管。"

"层层上报。逐级授权。下级服从上级。上级覆盖下级。"

周明忽然插了一句："那如果所有层级都说'我管不了'呢？"

"那就是你昨天看到的那个错误。"马农说。

他在白板上写：

`
ClassNotFoundException
`

"三个ClassLoader都找不到那个类。所以抛异常。"

"就像——"赵天想了想，"就像一份文件从士兵一路交到将军，将军也说'这不归我管'，然后整个系统就卡住了？"

"对。"马农说。"所以你的代码必须确保需要的类在classpath里。否则就是——请求到了最高层，最高层说不认识。整条链断裂。"

"现在回到昨天的问题。"马农擦掉了一部分白板，画了一个新的图：

`
Origin的钩子做了什么？

正常流程：
  App ClassLoader → 找 Hello.class → 加载 → 运行

Origin钩子流程：
  App ClassLoader → 先加载 RuntimeOptimizer → 找不到！
  → ClassNotFoundException → 你的程序还没开始就崩了
`

"Origin在2036年改了JVM的启动参数。它在classpath里注入了一个额外的加载器——OriginClassLoader。这个加载器会在你的代码运行之前，先尝试加载RuntimeOptimizer。"

"CHEP之前，这个类由Origin的本地代理提供。CHEP之后，代理死了。但启动参数还在操作系统的环境变量里。所以每次你运行Java程序，JVM都会先去找一个已经不存在的类。"

"这就是昨天的报错。"

他在白板上写了解决方案：

`bash
方案1：绕过Origin的钩子
java -Djava.system.class.loader=default -cp . Hello

方案2：清除环境变量中的Origin残留
unset ORIGIN_JAVA_OPTS
unset ORIGIN_CLASSPATH
`

"我已经写了一个脚本。今天下课后拷给你们。运行一次就能把操作系统里的Origin残留参数全部清除。"

他在白板角落写了脚本名：

`bash
~/scripts/clean-origin-hooks.sh
`

赵天举手："所以Origin不只是一个应用？它改了操作系统层面的东西？"

"它改了一切。"马农的声音很平。"环境变量、系统服务、开机启动项、JVM默认参数、甚至内核模块。Origin的设计理念是——覆盖一切，接管一切。2036年以后，全球99%的Java进程都是通过Origin的定制JVM启动的。"

他停了一下。

"这就是为什么CHEP之后，几乎没有人能跑起一个干净的Java程序。不是JDK坏了——是Origin在JDK之上套了一层壳。壳碎了，但碎片还在。"

教室里安静了。

下午。

综合练习时间——写一个简易计算器。

马农在白板上给了需求：

`
简易计算器：
- 从命令行读取输入（格式：3 + 5）
- 支持加减乘除
- 除数为0时打印错误信息
- 输入"exit"退出
`

"用你们学的全部内容。变量、循环、条件、方法、异常处理。四十分钟。"

这是第一周的毕业作业。

三十五分钟后。

八个人完成了。刘韧和林夕是最早的——分别在二十分钟和二十五分钟。

马农挨个检查。

刘韧的代码干净利落。方法拆得好——parseInput、calculate、printResult三个方法各司其职。

"你的Java没忘干净。"马农说。

"被你这么一逼，全想起来了。"刘韧苦笑。

林夕的代码多了一点东西——她在calculate方法里加了一个边界检查：除数为零时，不是简单抛异常，而是返回Double.NaN——"Not a Number"，表示这个结果没有意义。这是一个老程序员才知道的技巧。

`java
static double calculate(double a, String operator, double b) {
    switch (operator) {
        case "+": return a + b;
        case "-": return a - b;
        case "": return a  b;
        case "/":
            if (b == 0) {
                System.out.println("错误：除数不能为零");
                return Double.NaN;  // 关键：不是抛异常，而是返回NaN
            }
            return a / b;
        default:
            System.out.println("错误：不支持的运算符 " + operator);
            return Double.NaN;
    }
}
`

马农看了一眼。

"Double.NaN。"他说。"你怎么知道这个？"

"昨天学异常处理的时候查的。"林夕说。"Not a Number。表示'这个结果没有意义'。"

"你在哪查的？Origin的文档都不能用了。"

"你给的那个JDK离线安装包里有Java文档。在docs文件夹里。"

马农看了她三秒。

他没有说话。但他在心里把林夕的评估从"可造之材"移到了另一个位置。

一个会自己翻文档的人。

在Origin时代，这种能力几乎绝种了——因为没人需要翻文档。你只需要问Origin就行。

最后一个完成的是产品经理——叫陈莉。她的代码有一个bug，Scanner没有close()导致资源泄漏。马农指出来之后，她花了五分钟修复。

"第一周结束了。"马农说。

他没有总结。没有鼓励的话。没有"大家辛苦了"。

他在白板上写了第二周的课程表：

`
第二周：面向对象
Day 8:  类与对象——概念
Day 9:  类与对象——实践
Day 10: 封装、继承、多态
Day 11: 接口与抽象类
Day 12: 集合框架（ArrayList、HashMap）
Day 13: 泛型基础
Day 14: 综合练习——学生管理系统重构
`

"第二周比第一周难三倍。"马农说。"如果你在第一周已经觉得吃力——"

他停了。

所有人以为他要说"那就放弃吧"或者"需要额外补习"。

"——那就多花两倍时间。"

他拿起外套。

"明天八点。"

人走了之后。903室。

马农在培训计划上画了第七天的勾。他在旁边写了第一周的总评：

`
第一周总评：
存活率：12/12（无人退出）
达标率：8/12（8人能独立完成综合练习）
突出表现：刘韧（基础扎实）、林夕（自学能力）
需要关注：陈莉（产品思维强但编程直觉弱）、赵天（习惯在改善中）
整体：比预期好。
`

"比预期好"——在马农的词典里，这已经是很高的评价了。他原本预计第一周会有3-4人退出。

他关上文件。

然后打开了另一个文件——那个他一直在维护的系统监控。

`
903系统运行状态（Day 22）:
- 在线用户：4,892
- 日消息量：42,000+
- 服务节点：3（路由/DNS/认证）
- 待建服务：18
- 可维护人员：1（马农）
``

一个人。维护三个服务。支撑近五千人。

如果他食物中毒了呢？如果他累倒了呢？如果有更紧急的事需要他离开呢？

这就是为什么他要教。

不是因为他喜欢教。

是因为他必须教。

深夜。马农在终端前。

他不是在写代码。他在翻一份旧文件——师父的笔记《手写的温度》。U盘里的原始文件。三万字。

他翻到了第十二页。师父写的一段话：

"教一个人写代码，不是把你会的东西塞给他。是把他不会的东西一层一层剥开，让他自己看到里面的结构。你能教的只有方法。理解必须是他自己的。"

马农看了这段话很久。

然后他关掉了文件。

窗外。深圳的天空有一点点不一样了——不是因为灯光回来了。是因为空气变了。九月末的深圳，闷热开始消退，偶尔有凉风。

二十二天了。

903室的灯还亮着。十五台服务器还在跑。白板上的双亲委派图还没擦。

马农在行军床上躺下。

明天教类和对象。

面向对象是编程思维的第一道分水岭。过了这道关的人会开始理解"代码不是一行一行写的，是一块一块搭的"。过不了的人——

他闭上了眼睛。

三十秒。呼吸平稳。
