第31章 伪共享之毒

你以为两个变量毫无关联。CPU缓存说：不好意思，你们住在同一个房间。

CHEP后第三十九天。

903系统出了一个诡异的性能问题。

老李在凌晨三点发了一条紧急消息：

``
马农，路由服务的CPU从22%飙到了87%。
但日志显示请求量没变——还是那个数。
不知道什么吃了CPU。
`

马农起床了。他在终端前坐下。

top命令显示路由服务进程占用了六个核心中的四个。jstack显示所有线程都在RUNNABLE状态——没有锁等待，没有IO阻塞。纯计算。

但业务量没变。

"如果请求量没变，但CPU上去了——说明每次请求消耗的CPU增加了。"马农说。

他打开了一个性能计数器工具——perf。这是Linux内核级的性能分析工具，可以看到CPU的微架构事件。

`bash
perf stat -p $(pgrep -f EmergencyRouter) -- sleep 10
`

结果出来了：

`
Performance counter stats:
  12,847,231,094  cycles
   3,912,445,012  instructions     # 0.30 insn per cycle
   1,876,234,567  cache-references
   1,123,445,678  cache-misses     # 59.87% of all cache refs
`

马农看到了那个数字。

59.87%。缓存未命中率。

"正常值应该在5%以下。"他说。"60%——CPU的缓存几乎完全失效了。"

他继续追查。

路由服务在上周做过一次修改——赵天在马农的指导下，给路由表加了两个计数器。一个记录路由查询次数（queryCount），一个记录路由更新次数（updateCount）。两个计数器分别被不同的线程更新。

`java
public class RouterStats {
    volatile long queryCount;   // 线程A频繁更新
    volatile long updateCount;  // 线程B频繁更新
}
`

看起来完全无害。两个独立的计数器。不同的线程操作不同的字段。没有锁竞争。没有数据竞争。

但CPU缓存说：不。

马农在白板上画了一个图。

他把编程班的学员叫来了——虽然是凌晨四点。林夕和刘韧赶到了。赵天也来了（毕竟是他写的代码）。

"CPU缓存行。"马农在白板上画。

`
CPU缓存结构：
┌─────────────────────────────────┐
│         Cache Line (64字节)       │
│  [queryCount 8字节][updateCount 8字节][...其他数据...]  │
└─────────────────────────────────┘
`

"CPU不是一个字节一个字节地从内存读数据的。它一次读64个字节——这叫一个缓存行。"

"queryCount和updateCount各占8字节。加起来16字节。它们在内存里是挨着的——因为它们在同一个对象里。所以它们大概率被加载到了同一个缓存行里。"

他在白板上画了两个CPU核心：

`
核心0（线程A）          核心1（线程B）
  ┌──────────┐           ┌──────────┐
  │ Cache L1  │           │ Cache L1  │
  │ [缓存行X] │           │ [缓存行X] │
  │ query=100 │           │ update=50 │
  └──────────┘           └──────────┘
`

"线程A在核心0上修改queryCount。核心0的缓存行X被标记为'已修改'。"

"但核心1也有缓存行X的副本——因为updateCount也在里面。根据CPU的缓存一致性协议（MESI协议），核心0修改了缓存行X，核心1的副本就失效了。"

"核心1下次要读updateCount的时候，必须从内存重新加载整个缓存行——即使updateCount根本没变。"

他在白板上画了箭头：

`
核心0修改queryCount → 缓存行X失效 → 核心1重新加载
核心1修改updateCount → 缓存行X失效 → 核心0重新加载
来回失效。每次修改都要重新加载。这就是"伪共享"。
`

"它们在逻辑上完全独立。但在物理上——它们共享了同一个缓存行。"

"这就是伪共享——false sharing。两个变量像是住在同一个房间的室友。一个人开灯，另一个人也被吵醒。"

"怎么修？"赵天的脸色很难看。他的代码造成了凌晨四点的紧急事件。

"把它们分开。"马农说。"让两个计数器不要住在同一个缓存行里。"

他在白板上写了修复方案：

`java
// 修复前：两个long在同一缓存行（16字节 < 64字节）
public class RouterStats {
    volatile long queryCount;
    volatile long updateCount;
}

// 修复后：用@Contended让JVM自动填充，隔离到不同缓存行
import jdk.internal.misc.Contended;

public class RouterStats {
    @Contended
    volatile long queryCount;
    
    @Contended
    volatile long updateCount;
}
`

"@Contended注解。"马农说。"JVM会在这个字段前后填充空字节，确保它独占一个缓存行。两个计数器不再共享缓存行——不再互相干扰。"

他在白板旁边写了运行参数：

`bash
@Contended需要这个参数才能生效
java -XX:-RestrictContended RouterService
`

"改完之后——重新编译、重启。"

五点。

修复部署了。

`
修复前：CPU 87%, cache-miss 59.87%
修复后：CPU 24%, cache-miss 3.12%
`

CPU从87%降到24%。缓存未命中从60%降到3%。

赵天看着屏幕上的数字。

"两个变量挨在一起……就能造成这么大的影响？"

"硬件不关心你的代码逻辑。"马农说。"它只关心内存布局。你写的代码在源码层面可能完美。但在CPU层面——两个'无关'的变量如果碰巧住在同一行——性能就崩了。"

他看着赵天。

"这不是你的错。你写计数器的逻辑是对的。命名规范也对了。但性能问题往往不在逻辑层——在物理层。"

赵天低着头。

"记住一件事。"马农说。"你调查bug的速度——从发现到修复——决定了系统能活多久。今天这个bug如果拖到早上八点再修——七千人在七千用户高峰期面对一个87%CPU的系统——会怎样？"

赵天想了想："系统变慢。消息发不出去。有人会以为系统崩了。"

"对。所以你学到的不只是伪共享。是——凌晨四点的告警要在凌晨四点处理。不是明天早上。"

那天上午。马农把这个案例写进了教学笔记。

`
Day 25 案例：伪共享（False Sharing）

现象：CPU飙升但请求量不变
根因：两个volatile long字段在同一缓存行，不同线程交替修改导致缓存行失效风暴
修复：@Contended注解隔离缓存行
教训：
  1. 性能问题的根因往往在你看不到的层面（硬件/OS/JVM）
  2. 两个"无关"的变量可能在物理上高度耦合
  3. volatile不是免费的——它的代价隐藏在缓存一致性协议里
`

然后他打开了手册v1.0。在第三部分"多线程基础"后面加了一节：

`
3.6 伪共享——CPU缓存行的暗毒

如果你的多线程程序莫名其妙地慢，而锁和逻辑都没问题——
检查你的热点变量是不是挨在一起。

CPU缓存行=64字节。
两个变量如果在同一个64字节范围内，会被CPU一起加载。
一个线程改了其中一个，另一个线程的缓存就失效了。

解决：@Contended注解，或手动填充padding字段。

这是性能优化里最难发现的问题之一。
因为代码看起来完全没问题。问题在代码看不到的地方。
`

他保存了手册v1.1。

下午。赵天找到马农。

"马农。"他说。"对不起。"

马农看了他一眼。

"你道歉的对象不是我。是凌晨三点被吵醒的老李。"

赵天点头。他去找了老李。

后来老李跟马农说："赵天给我道了歉。还给我泡了一杯速溶咖啡。"

马农说："嗯。"

但他在人事评估文件里加了一行：

`
赵天——Day 25事件后，主动道歉+反思。
从Day 3的命名问题到Day 25的伪共享事件。
他在变好。速度比我预期的快。
``
