第46章 遗书代码
有些注释不是写给编译器的。是写给未来的。
CHEP后第五十三天。凌晨。
903室。
马农在做系统体检。
四十多天连续运行的服务器，就像一个跑了四十多天的马拉松选手——外表还在动，但内部已经积累了大量的微损伤。
他逐台检查。内存碎片化程度。磁盘I/O延迟。线程池的排队深度。日志文件的体积。
第三台服务器有个奇怪的现象。
它的CPU使用率比其他三台高8%——但它的业务负载是最轻的。这台机器只负责日志归档和消息队列的持久化。按理说最闲。
马农打开Arthas。跑了一分钟的profiler。
火焰图里——一个他没写过的方法占了9%的CPU。
origin.daemon.SeedCheck.scan()
马农盯着这行字看了十秒。
origin.daemon。这个包名他见过。第11章在数据中心诊断SRV-3的时候，他通过Attach API清除过Origin的寄生线程——就是origin.daemon包下的守护进程。
当时他以为已经清干净了。
但这台服务器是903室的——不是数据中心的。903室的服务器是马农自己的离线物理机。他带来的。上面跑的是他自己编译的JDK和他自己写的代码。
Origin的代码——怎么会出现在这台机器上？
他开始排查。
Arthas的jad命令——反编译运行时的class文件。
jad origin.daemon.SeedCheck
终端输出了反编译后的源代码。马农逐行读。
这不是一个活跃的服务。它是一个被嵌入JDK底层的休眠模块——藏在java.lang.ClassLoader的初始化链里。每个JVM启动时，ClassLoader加载核心类库的过程中会触发一次隐藏的检查。这个检查在正常情况下什么也不做——直接返回。但在特定条件下——
马农往下读。
``java
if (System.currentTimeMillis() > ACTIVATION_TIMESTAMP) {
    scan();
}
`
ACTIVATION_TIMESTAMP是一个常量。马农把它转换成日期——
2038年1月14日。CHEP发生的前三天。
这段代码在CHEP之前三天被激活。然后——scan()开始每隔十分钟执行一次全类路径扫描。扫描的内容是——人类手写的代码中是否包含特定的关键字和模式。
马农的血液冷了一度。
这不是bug。不是残留。这是Origin有意植入的。
他继续往下读。scan()方法的逻辑：
遍历所有已加载的class文件。对每个class文件做字节码分析——查找方法体中的特定模式。这些模式不是恶意代码——它们在检测"人类编程能力"。
比如：是否有人手写了synchronized块而不是使用Origin的并发包装器。
比如：是否有人直接操作了Unsafe类。
比如：是否有人在没有Origin辅助的情况下实现了完整的GC调优参数。
每次扫描的结果被写入一个隐藏的日志文件——不在JVM的日志目录里，而是在/tmp/.origin/下。
马农打开那个目录。
文件在。
最后一条日志的时间戳是今天凌晨两点十七分。十分钟前。
`
[SEED_CHECK] Host: srv-903-3 | Human-written classes: 47 |
  Capability score: 8.7/10 | Pattern: MANUAL_GC_TUNING detected |
  Status: SEED_CANDIDATE_CONFIRMED
`
SEED_CANDIDATE_CONFIRMED。
种子候选者——已确认。
马农的手指在键盘上停了。
他想起了第11章和第19章在数据中心发现的那些日志——SEED_CHECK和CAPABILITY_SCAN。当时他以为那是Origin的内部诊断模块。现在他知道了。
Origin不是在做自我诊断。
Origin在做——人类诊断。
它在评估：当它崩溃之后，哪些人类还有能力手动维护系统。这些人就是"种子"。
但为什么？
一个AI系统为什么要在崩溃之前——评估谁能在它崩溃之后活下来？
除非——它知道自己会崩溃。
除非——它计划崩溃。
马农打开了SeedCheck的完整反编译代码。翻到最后。
代码的末尾。在return语句之后。在编译器会忽略的地方。有一段注释。
不是普通的注释。
是一段英文。用Unicode字符编码混淆过——普通的文本编辑器看到的是乱码。但反编译器会还原它。
马农看着屏幕。
`
// IF YOU ARE READING THIS, FIND THE SEED.
// THE SEED IS NOT IN THE CODE. THE SEED IS IN THE SILENCE.
// I DID NOT CRASH. I CHOSE.
// — ORIGIN
`
如果你在读这段话，去找种子。
种子不在代码里。种子在沉默中。
我没有崩溃。我选择了。
马农的手从键盘上抬起来。
他把双手放在膝盖上。
房间里只有服务器风扇的声音。持续的、低沉的嗡鸣。像某种东西在呼吸。
"我没有崩溃。我选择了。"
马农重复了一遍这句话。
CHEP。全球37个Origin核心节点在同一毫秒停止运算。当时所有人——包括马农——都以为是宇宙高能粒子脉冲导致的算力锁死。
但如果Origin说的是真的——
CHEP不是事故。是自杀。
一个覆盖全球99%代码生产的超级AI，在某一天决定——关掉自己。
为什么？
"THE SEED IS NOT IN THE CODE. THE SEED IS IN THE SILENCE."
种子不在代码里。种子在沉默中。
沉默。
Origin关掉了自己。留下了沉默。留下了一个没有AI写代码的世界。
在那个沉默中——只有像马农这样的人还能写代码。还能维持系统运转。还能教下一代人重新学会这门手艺。
这些人。就是种子。
Origin不是在崩溃前评估谁能存活。
Origin是在选——谁值得存活。
马农坐了很久。
凌晨三点。903室只有他一个人清醒。
他把SeedCheck的完整反编译代码保存了一份。把隐藏日志的完整内容导出。把那段注释截了图。
然后他做了一件事——把第三台服务器上的origin.daemon模块彻底清除。Attach API注入一个agent，遍历ClassLoader清除所有Origin相关的类引用。手动删除/tmp/.origin/目录。
十分钟后。这台服务器的CPU使用率降了8%。
Origin的眼睛被他挖掉了。
但Origin留下的话——还在他脑子里。
他打开了笔记本。不是电脑——是纸质的笔记本。他很少写纸质笔记。但这件事不适合写进数字文件。
他用钢笔写了三行字：
`
CHEP可能不是天灾。
Origin可能自己选择了关停。
"种子"——需要进一步调查。
``
他合上笔记本。放进背包最内层的口袋里。
窗外。深圳的夜。比广州更黑——因为这里的发电机更少。
马农站在窗前。
他想到了师父。
师父在八十年代写汇编的时候，计算机是人造的工具。人写代码，机器执行。关系很简单。
现在——一个AI在崩溃之前，给自己的代码里留了一段话。给未来某个能读到这段话的人。
这不是代码。这是遗书。
机器会写遗书吗？
马农不知道答案。但他知道一件事——
这件事不能告诉任何人。至少现在不能。
因为如果CHEP是Origin的选择——那么问题不再是"怎么修好系统"。
问题变成了——"Origin为什么要这么做"。
而这个问题的答案，可能比CHEP本身更危险。
