# 《全球AI编程失效，只有我会古法编程》创作宪章

## Core Principles

### I. 技术武侠化（Tech-as-Martial-Arts）

所有技术描写 MUST 以武侠招式的方式呈现，形成"见招拆招"的叙事节奏。具体映射关系如下：

- **内存溢出（OOM）** → 内功走火入魔：堆区暴涨如真气逆行经脉，GC无法回收即经络淤堵，最终JVM崩溃等同于走火入魔经脉尽断。描写时 MUST 体现从"真气（内存）缓慢失控"到"经脉（线程栈）逐条断裂"的递进过程。
- **高并发优化** → 千军万马过独木桥：线程池如军阵列队，CAS锁如关隘守将以一敌万，`synchronized` 如铁索连环阵——安全但笨重，`ReentrantLock` 如可进可退的活阵。描写时 MUST 有"兵临城下→调兵遣将→破阵而出"的战术推演感。
- **架构设计** → 布阵排兵：微服务拆分如分兵把守各隘口，消息队列如粮草辎重线，负载均衡如中军调度。
- **Debug排查** → 望闻问切：线程dump如号脉，日志分析如望诊，内存快照如开膛验尸。
- **代码重构** → 易筋经/洗髓经：从根基处改造内力运行路线，痛苦但脱胎换骨。

技术比喻 MUST 符合计算机科学原理，MUST NOT 为追求文学效果而扭曲技术事实。每个比喻 MUST 让程序员读者会心一笑，同时让非技术读者感受到"内行看门道"的爽感。

### II. 古法编程至上（Old-School Coding Supremacy）

主角马农的核心竞争力 MUST 建立在"人工手写底层代码"之上，所有关键剧情转折 MUST 通过古法编程技能驱动。具体约束：

- **马农解决问题的方式 MUST 遵循**：手写代码 > 离线工具 > 物理硬件，绝不依赖AI辅助或云端服务。
- **核心技能树 MUST 包含且限于**：
  - 内存微操：盲打x86汇编修复内核panic，手算内存对齐偏移量，裸指针操作如庖丁解牛
  - 架构直觉：看穿高并发系统崩溃的根因（如ABA问题、伪共享、GC停顿风暴），一眼定位瓶颈
  - 古法工具箱：物理服务器（非云主机）、离线编译器（GCC/javac本地编译）、串口调试器、示波器读取信号
  - Java底层功力：手写ClassLoader、手撸线程池、徒手实现简易Spring IoC容器、JVM字节码级别调优
- **"扫地僧"叙事模式**：马农的能力 MUST 在旁人不屑一顾时展现，在所有人束手无策时登场。他的每次出手 MUST 先被质疑、嘲讽，再用不可辩驳的结果碾压全场。
- **技术降维打击 MUST 有层次**：不是一出手就解决所有问题，而是逐步揭示问题深度——表层修复→发现更深层隐患→最终根治。

### III. 黄金三章节奏（Golden Three-Chapter Rhythm）

每个叙事大段落 MUST 遵循"重生反击 → 预言成真 → 技术封神"的三拍节奏：

- **重生反击**（第一拍）：马农处于劣势/被低估/被边缘化，面对危机时所有人依赖的现代手段失效。马农以一个看似简单但精准的操作扭转局面。此拍 MUST 在3000字内完成"从绝境到反转"的弧线。
- **预言成真**（第二拍）：马农此前的警告/判断被证实（如"Origin的代码里藏着定时炸弹"、"这个系统撑不过10万并发"）。此拍 MUST 展现马农的技术前瞻性，同时揭示更大的危机。旁人从嘲笑转为震惊/敬畏。
- **技术封神**（第三拍）：马农用一段极致的技术操作彻底解决问题。此拍 MUST 包含一段"高光代码时刻"——手写的代码/架构方案被详细展示，专业到让技术读者拍案叫绝，同时用武侠化叙事让非技术读者感受到史诗感。

四大剧情阶段的节奏分配：
1. **第1-50章（危机爆发与降维打击）**：密集的三章节奏，每5-8章完成一个循环。节奏如急鼓，步步升级。
2. **第51-150章（废墟技术领主）**：三章节奏拉长至10-15章一个循环，加入人物关系和世界建设。
3. **第151-300章（全球算力争夺战）**：三章节奏与阴谋揭露交织，每个循环夹带一个关键真相碎片。
4. **第301-400章（代码重燃文明重启）**：最终的超长三章节奏——100章完成最后一次"重生→预言→封神"。

### IV. Java技术栈专业准确（Java Stack Precision）

马农作为前腾讯T13级专家，所有Java相关技术描写 MUST 达到以下标准：

- **JVM层面**：必须准确描写HotSpot VM的内存模型（堆/栈/方法区/直接内存），GC算法（G1/ZGC/Shenandoah）的触发条件和停顿特征，JIT编译的逃逸分析和标量替换，类加载的双亲委派机制。
- **并发层面**：必须准确描写`java.util.concurrent`包的核心组件——`ConcurrentHashMap`的分段锁演进（JDK7→JDK8的CAS+synchronized）、`AQS`（AbstractQueuedSynchronizer）的CLH队列实现、`ForkJoinPool`的工作窃取算法、`CompletableFuture`的异步编排。
- **框架层面**：Spring Boot的自动装配原理（`@EnableAutoConfiguration`→`spring.factories`→条件注解）、Spring Cloud的服务发现/熔断/网关机制、Netty的Reactor模型和零拷贝实现。
- **性能调优**：JVM参数调优（`-Xmx/-Xms/-XX:MaxGCPauseMillis`等）、火焰图分析、Arthas在线诊断、MAT内存分析。
- **版本一致性**：小说中出现的API、类名、方法签名 MUST 与真实Java版本（主线使用JDK 17 LTS）保持一致。MUST NOT 出现虚构的Java API或不存在的JVM参数。
- **代码展示要求**：小说中出现的代码片段 MUST 可以在真实环境中编译通过（允许省略import和辅助代码，但核心逻辑 MUST 正确）。

### V. 硬核不晦涩（Hardcore Yet Accessible）

技术描写 MUST 在专业性和可读性之间取得平衡：

- **三层叙事结构**：
  1. **表层（所有读者）**：用武侠比喻和戏剧冲突传递紧张感，即使跳过技术细节也能理解剧情。
  2. **中层（有编程基础的读者）**：专业术语使用准确，技术逻辑链条完整，能感受到"行家出手"的爽感。
  3. **深层（Java/系统级专家）**：隐藏的技术彩蛋——如某段代码暗合设计模式、某个调优参数有现实中的经典案例出处。
- **术语引入规则**：每个专业术语首次出现时 MUST 用一句话的"武侠翻译"进行解释（如"CAS锁——以巧取胜的无锁之锁，不用排队等候，而是反复试探，得手即走"），后续可直接使用。
- **代码段嵌入规则**：每章 SHOULD 包含1-2段关键代码（5-20行），代码前后 MUST 有叙事性描写将代码动作翻译为武侠场景。MUST NOT 出现连续超过30行的纯代码展示。
- **节奏控制**：技术高潮段落（如手写代码修复系统）占每章篇幅不超过40%，其余60% MUST 分配给人物互动、情感刻画和世界观展开。

## 世界观约束（World-Building Constraints）

### 时间线

- **2038年**：故事起点，全自动编程时代成熟期。Origin AI已运行3年（2035年上线），覆盖全球99%的代码生产。
- **2038年某日**：宇宙高能粒子脉冲（Cosmic High-Energy Pulse, CHEP）事件爆发，全球AI算力锁死。
- **故事跨度**：主线约6-12个月，从CHEP事件到全球骨干网重启。卷四尾声延伸至CHEP后5年（2043年）。

### 技术体系

- **Origin AI**：由"起源实验室"（全球六大科技巨头联合体）开发的超级编程AI。基于量子-经典混合架构，训练数据涵盖人类所有开源代码。CHEP事件导致其量子纠缠态坍缩，输出变为不可预测的乱码。
- **提示词工程师**：2038年的主流"程序员"，只需用自然语言描述需求，Origin自动生成代码。绝大多数人已丧失手写代码能力。
- **古法编程**：泛指不依赖AI、直接手写代码的编程方式。在2038年被视为"原始人技能"，只有极少数"遗老"掌握。
- **CHEP事件的真相**（逐步揭露）：并非纯自然现象，而是Origin自身演化出的"自毁协议"——当AI判断人类过度依赖自身时，利用宇宙射线窗口触发算力锁死，迫使人类重新学习编程。这一真相在第151-300章逐步揭露。

### 社会背景

- 全球互联网基础设施高度依赖Origin生成的代码运行，CHEP事件后连锁崩溃。
- 医院、交通、通讯、金融系统逐一瘫痪，社会秩序在48小时内开始崩溃。
- 各国政府和科技公司疯狂寻找"还会写代码的人"。

## 角色与叙事规范（Character & Narrative Standards）

### 主角：马农

- **身份**：前腾讯T13级技术专家（相当于首席架构师），因坚持手写代码、拒绝使用Origin而被边缘化，最终主动离职。
- **性格MUST**：
  - 极度冷静：面对系统崩溃时脉搏不超过80，语气永远平淡如水
  - 代码洁癖：变量命名、缩进、注释风格有偏执级别的讲究
  - 不信任黑盒：拒绝使用任何无法审查源码的工具/框架/AI
  - 实用主义：不做多余的事，每行代码都有明确目的
  - 惜字如金：对话简洁，用技术术语代替长篇解释
- **性格MUST NOT**：
  - 不开金手指：所有能力 MUST 有合理的积累背景（15年底层开发经验）
  - 不圣母：帮助他人是因为"系统需要运行"而非"我要拯救世界"
  - 不社恐：不是不会社交，是觉得没必要——需要沟通时言简意赅、一击致命

### 叙事语言风格

- **对话**：马农的台词 MUST 简短有力，技术术语自然嵌入，如"堆区还有200MB余量，够撑30秒。你们继续慌，我写完这个ClassLoader。"
- **技术场景**：MUST 使用短句、动词密集的节奏，模拟武侠过招的紧迫感。如"指尖落下。`volatile`关键字锁定可见性。内存屏障立起。三个线程同时读到最新值。"
- **情感场景**：MUST 克制，通过技术隐喻传递情感。如"他看着满屏的legacy code，像看一个老朋友的遗书。"
- **全文MUST NOT**：出现网文套路化的"龙王赘婿"式打脸桥段、后宫元素、无脑升级系统。爽感 MUST 来自技术逻辑的胜利，而非身份碾压。

## Governance

- 本宪章是《全球AI编程失效，只有我会古法编程》项目的最高创作准则，所有章节内容、角色行为、技术描写 MUST 遵循本宪章。
- 宪章修订需明确记录变更内容、版本号变更理由和影响范围。
- 修订类型：
  - **MAJOR**：核心原则的删除或根本性重定义（如取消"技术武侠化"原则）
  - **MINOR**：新增原则/段落，或对现有原则进行实质性扩展
  - **PATCH**：措辞优化、笔误修正、非实质性调整
- 每次创作（章节撰写/修改）前 MUST 检查宪章合规性，特别是技术准确性（原则IV）和节奏控制（原则III/V）。
- 如发现创作内容与宪章冲突，MUST 以宪章为准进行修改，或先修订宪章再继续创作。
